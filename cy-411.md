## 项目主要需求

### 1. 项目概述

**碳足迹追踪平台（CarbonTrack）** 是一款面向个人与小微企业的碳排放记录与分析系统。用户可记录日常出行、能源消耗、餐饮消费等行为，系统自动计算碳排放量，生成趋势图表、减排建议与排行榜。项目类型：**全栈 Web 应用**。

### 2. 核心实体（≥ 3 个，必须贯穿全栈）

- **User（用户）**
  - 字段：id, username, email, password_hash, avatar, region, created_at
  - 关键关系：一个用户拥有多条 Activity 记录和多条 Goal 记录
  - 业务动作：注册、登录、修改资料、查看个人碳排趋势
  - **此实体必须贯穿：数据库表 → backend/models/user.js → backend/services/userService.js → backend/controllers/userController.js → backend/routes/users.js → frontend/src/api/user.ts → frontend/src/stores/userStore.ts → frontend/src/pages/Profile.tsx**

- **Activity（碳排放活动）**
  - 字段：id, user_id, category（交通/能源/餐饮/购物）, sub_type, amount, unit, carbon_value, record_date, note
  - 关键关系：隶属于 User，引用 Category 的排放因子
  - 业务动作：新增记录、编辑、删除、按日期/分类筛选
  - **此实体必须贯穿全栈，禁止在单一文件中处理 CRUD**

- **Goal（减排目标）**
  - 字段：id, user_id, title, target_value, period_type, start_date, end_date, status
  - 关键关系：隶属于 User，与 Activity 的汇总数据联动
  - 业务动作：设定目标、查看完成进度、到期提醒
  - **此实体必须贯穿全栈**

- **CarbonFactor（碳排放因子）**
  - 字段：id, category, sub_type, factor_value, unit, region, updated_at
  - 关键关系：被 Activity 引用以计算 carbon_value
  - 业务动作：后台管理因子、按地区匹配
  - **此实体必须贯穿全栈**

### 3. 核心页面（≥ 4 个，至少 2 个页面共用同一业务组件）

- **/dashboard（仪表盘）**
  - 消费实体：User、Activity、Goal
  - 关键交互：展示今日/本周/本月碳排放总量、趋势折线图、目标完成进度条
  - 共享组件：`<CarbonTrendChart>`、`<GoalProgressCard>`

- **/activities（活动记录）**
  - 消费实体：Activity、CarbonFactor

- **/goals（目标管理）**

  - 关键交互：创建目标、查看进度、到期提醒
  - 共享组件：`<GoalProgressCard>`（与 /dashboard 共用）

- **/ranking（排行榜）**

  - 关键交互：按地区/时间段查看用户排名

  - 共享组件：`<AvatarUploader>`

### 11. 特殊约束：屎山代码设计（低内聚、高耦合、牵一发动全身）

为了验证模型的跨文件协同改动能力，本项目在规范分层的基础上，额外要求刻意制造合理的"屎山"耦合：

1. **日志模块必须单独管理但全栈引用**
   - 后端必须有一个 `utils/logger.js`，使用 `winston` 或自建日志类
   - 所有 controller、service、middleware 都必须引用 logger
   - 日志格式字符串必须集中定义在 `constants/logTemplates.js`，包含至少 20 条模板，如 `USER_LOGIN_SUCCESS`、`ACTIVITY_CREATE_FAILED`
   - 任何业务字段变更（如 Activity 新增字段）都必须同步修改对应的日志模板和调用处

2. **异常信息必须分散且层层透传**
   - 错误码集中在 `constants/errorCodes.js`，但每个 service/controller 都要手动拼接 message
   - 禁止在单一地方统一处理所有异常，必须在 controller 中再次包装 service 抛出的错误
   - 错误 message 中必须包含实体名和字段名，如 `"Activity[id=123] create failed: category invalid"`

3. **常量/工具类必须多处耦合**
   - `utils/formatters.ts` 中同时包含日期格式化、金额格式化、碳排放格式化、状态文本映射
   - 多个组件和页面都直接引用 formatters，而不是独立拆分
   - `constants/messages.ts` 同时包含前端提示文案、后端返回文案、日志文案
   - 修改一个文案可能同时影响前端显示、后端返回和日志内容

4. **全局配置必须通过多个文件读取**
   - `config/app.ts` 读取环境变量，但 controller/service 中也可以直接读取 `process.env`
   - 任何新增配置项都需要在 config、.env.example、docker-compose.yml、utils 中同步出现

5. **枚举必须前后端重复定义且被多处引用**
   - ActivityCategory 和 GoalStatus 必须在前端 constants、后端 constants、类型定义、日志模板、错误码中同时存在
   - 新增一个枚举值时，必须修改至少 8 处文件

6. **牵一发动全身的验证标准**
   - 后续如果要求"给 Activity 新增一个 'waste' 分类"，必须触达：数据库 migration、Activity 实体、CarbonFactor 实体、前后端 constants、日志模板、错误码、formatters、ActivityCard 组件、筛选器、Dashboard 图表分类等 ≥ 8 个文件

### 12. 部署要求

这是前后端分离项目。

前端代码放在 `frontend/`：
- 页面、组件、路由、状态管理、API 请求、类型定义、样式和资源等放在对应的子文件夹

后端代码放在 `backend/`：
- 数据库脚本放在 `database/`。

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

需要提供以下文件：
- 根目录 `docker-compose.yml`：编排前端、后端、数据库及所有依赖服务
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=carbontrack`
- `frontend/Dockerfile`：多阶段构建，先构建再用 Nginx 托管
- `backend/Dockerfile`：后端项目构建与运行镜像
- `frontend/nginx.conf`：Nginx 配置，处理前端路由和 API 反向代理
- `.env.example`：所有环境变量的示例文件，附注释说明；同时包含 `COMPOSE_PROJECT_NAME` 默认值

docker-compose.yml 要求：
- **不要写 `version:` 字段**
- 顶层声明 `name: carbontrack`
- 所有服务使用 `container_name` 命名容器，容器名带 `${COMPOSE_PROJECT_NAME:-carbontrack}` 前缀
- 数据库数据使用命名卷持久化（`volumes:` 顶层声明，不要使用绑定挂载到中文路径）
- 前端通过 Nginx 反向代理后端 API（前端代码里调用 `/api`，由 Nginx 转发到 `backend:<port>`，不要硬编码 `localhost`）
- 服务间通过 Docker 内部网络通信，服务名即主机名
- 数据库服务配置 `healthcheck`，后端服务通过 `depends_on: { db: { condition: service_healthy } }` 等待数据库就绪
- 后端服务也配置 `healthcheck`，前端 `depends_on` 后端健康
- 暴露端口：前端 `18411:80`，后端 `19411:3000`，数据库 `3306:3306`
- 所有可变配置通过环境变量注入，并在 `.env.example` 中给出默认值
- 支持通过 `docker compose up -d` 一条命令启动全部服务

`.env` / `.env.example` 必须包含的键：
- `COMPOSE_PROJECT_NAME=carbontrack`
- 数据库连接相关：`DB_NAME=carbontrack_db`、`DB_USER=carbontrack_user`、`DB_PASSWORD=carbontrack_pwd`、`DB_ROOT_PASSWORD=carbontrack_root`
- 后端：`JWT_SECRET=change_me_to_a_long_random_string`
- 端口映射：`FRONTEND_PORT=18411`、`BACKEND_PORT=19411`、`DB_PORT=3306`

Nginx 反向代理要求（`frontend/nginx.conf`）：
- `location /api/` 反向代理到 `http://backend:3000/`
- 配置 `try_files $uri $uri/ /index.html;` 支持前端 SPA 路由
